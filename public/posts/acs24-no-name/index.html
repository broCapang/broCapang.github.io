This was a challenge similar to the one Jeremy created for Battle of Hackers 2024 so we solved it relatively fast. The challenge provides a binary that presents a menu that allows the user to borrow a loan, repay the loan, mining for money, buy a name and change name.

The program basically is a program where you can Loan Repay Mining to get money Buy Name Change Name Exit

Loan: Borrow Money, then increase debt Repay: Pay Debt Mining: Get money but only 0.01*5 max each time, and it will take longer and longer Buy Name: To get ourself a name, but cannot change name if got debt Change Name: To change our name Exit: exit the program

The goal is to get a name. But

  • To buy a name, you need to have money.
  • To have money you cannot simply mine, cause it will take a lot of time.
  • So you need to loan and then repay them.

The vulnerability lies in the way the program keeps track of the user’s loan.

But :-

To buy a name, you need to have money. To have money you cannot simply mine, cause it will take a lot of time. So you need to loan and then repay them.

First Vulnerability

MinerAccount Struct


struct MinerAccount {
	float cash;
	float debt_balance;
	int mining_attempts;
	char name[0x20];
};

Loan function

void loan(struct MinerAccount *account) {
	uint32_t amount = 0;
    
	printf("How much loan would you like to request?\n");
	if(scanf("%d", &amount) != 1) {
    	printf("Invalid input\n");
    	return;
	}
    
	if(account->debt_balance + amount > MAX_LOAN) {
    	printf("Loan limit exceeded\n");
    	return;
	}

	account->cash += amount;
	account->debt_balance += amount;

	printf("Current cash: $%.2f\n", account->cash);
	printf("Debt balance: $%.2f\n", account->debt_balance);
}

Notice how the debt in MinerAccount is FLOAT and in Loan function is INT. This will cause floating point inaccuracy vulnerability. A float is 32 bit and but it has 1 bit for sign, 23 bit for mantissa and 8 bit for exponent. Hence it can only print integers accurately up to 2^24 (16777216). If we go past this, it will start to display integers inaccurately. For example,

16777216 + 1 will result in 16777216

So with this vulnerability, we can borrow loan up to 2^24, then buy the name and take loan of size $1 which will increase our money but our debt wont increase.

Second Vulnerability

#define MAX_BUF 0x200
struct MinerAccount {
	float cash;
	float debt_balance;
	int mining_attempts;
	char name[0x20];
};

void change_name(struct MinerAccount *account) {
	if (has_name_rights != 1) {
    	printf("You do not have the right to change your name.\n");
    	printf("Please purchase a name to gain the rights to rename your no-name.\n");
    	return;
	}
	if(account->debt_balance != 0) {
    	printf("You still have debts to repay.\n");
    	printf("Pay off your debts to rename your no-name.\n");
    	return;
	}
	printf("Enter new name.\n");
	read(0, account->name, MAX_BUF);

	printf("Name updated successfully.\n");
}

The name in MinerAccount object were assigned to only 0x20 size, but in change_name function we can change up until 0x200. Buffer Overflow occur here.

Third Vulnerability

int main() {
	initialize();
	srand(time(NULL));
	struct MinerAccount account = {0, 0, 0, "no-name"};
	while(1) {
    	int choice;
    	printf("===========================\n");
    	printf("Welcome to %s\n", account.name);
    	printf("Current cash: $%.2f\n", account.cash);
    	printf("Debt balance: $%.2f\n", account.debt_balance);
    	printf("===========================\n");

    	printf("1. Loan\n2. Repayment\n3. Mining\n4. Buy Name\n5. Change Name\n6. Exit\nChoose an action.\n");
    	scanf("%d", &choice);
    	switch(choice) {
        	case 1:
            	loan(&account);
            	break;
        	case 2:
            	repayment(&account);
            	break;
        	case 3:
            	mining(&account);
            	break;
        	case 4:
            	buy_name(&account);
            	break;
        	case 5:
            	change_name(&account);
            	break;
        	case 6:
            	return 0;
        	default:
            	printf("Invalid choice\n");
            	break;
    	}
	}
	return 0;
}

In main function, account.name will be printed out with ‘%s’ specifier, means it will print out all characters up until it reaches Null Byte. This can be exploited to leak addresses and canary

Exploitation Process

We loan 16777216 money Then we buy name so our money no 16777216 - 1337 Then if we loan 1 dollar each time, our cash increase, but debt stays the same. So we loan 1 dollar for 1337 times Then can repay all debt Now start the leaking process through name Leak canary Leak libc_start_main address Proceed will rop chain to system
